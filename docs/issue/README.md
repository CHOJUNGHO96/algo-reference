# Issue 문서 관리 가이드

이 폴더는 프로젝트 개발 중 발생한 이슈와 해결 방법을 체계적으로 기록하고 관리합니다.

```
issue/
├── README.md           # 이 파일
├── 양식.md             # 이슈 기록 템플릿
├── database/           # 데이터베이스 관련 이슈
├── api/                # API 관련 이슈
├── frontend/           # 프론트엔드 관련 이슈
├── backend/            # 백엔드 관련 이슈
├── devops/             # 배포/인프라 관련 이슈
├── security/           # 보안 관련 이슈
├── performance/        # 성능 관련 이슈
└── misc/               # 기타 이슈
```

## 📝 이슈 기록 규칙

### 파일명 형식

```
날짜_이슈내용짧게요약.md
```

**예시**:
- `20260212_alembic_connection_error.md`
- `20260213_cors_policy_block.md`
- `20260214_memory_leak_fix.md`

### 파일명 작성 가이드

1. **날짜**: `YYYYMMDD` 형식 (8자리)
2. **이슈 요약**: 영문 소문자, 언더스코어(`_`)로 단어 구분
3. **길이**: 가능한 짧고 명확하게 (3-5 단어 권장)
4. **동사 사용**: 문제 상황이나 해결 내용을 나타내는 동사 사용
   - `fix`, `error`, `issue`, `problem`, `fail` 등

## ✍️ 이슈 작성 방법

### 1. 템플릿 복사

```bash
cp issue/양식.md issue/[카테고리]/[날짜]_[이슈요약].md
```

### 2. 필수 작성 섹션

#### 메타데이터
- **날짜**: 이슈 발생 또는 해결 날짜
- **카테고리**: 위의 카테고리 중 하나 선택
- **심각도**: Critical | High | Medium | Low
- **상태**: 해결됨 | 진행중 | 보류 | 미해결

#### 문제 상황
- 에러 메시지 (있는 경우)
- 재현 방법
- 증상
- 발생 환경

#### 원인 분석
- 근본 원인 (한 문장 요약)
- 상세 분석
- 잘못된 가정 또는 설정

#### 해결 방법
- 개요
- 단계별 해결 과정
- 코드 변경 사항

#### 검증
- 테스트 방법
- 검증 결과
- 검증 체크리스트

### 3. 권장 작성 섹션

- **교훈**: 이 이슈에서 배운 점
- **재발 방지 방안**: 동일한 문제가 재발하지 않도록 하는 방법
- **관련 이슈**: 연관된 다른 이슈 링크
- **참고 자료**: 도움이 된 문서나 링크

## 🎯 작성 팁

### 좋은 이슈 문서의 특징

1. **명확한 제목**: 문제가 무엇인지 한눈에 알 수 있음
2. **재현 가능**: 다른 사람이 같은 문제를 재현할 수 있음
3. **근본 원인 분석**: 표면적 증상이 아닌 근본 원인 파악
4. **검증 가능**: 해결 방법이 실제로 효과가 있음을 증명
5. **학습 중심**: 이슈에서 무엇을 배웠는지 명확히 기록

### 작성 시 주의사항

- ❌ 민감한 정보 (비밀번호, API 키 등) 포함 금지
- ❌ 너무 간략한 설명 지양 (미래의 나를 위해 상세히)
- ❌ 해결되지 않은 이슈를 해결됨으로 표시 금지
- ✅ 에러 메시지는 전체를 복사
- ✅ 코드 변경 사항은 변경 전/후 비교
- ✅ 스크린샷이나 로그 파일 첨부 고려

## 🔄 이슈 상태 관리

### 상태 전환 흐름

```
🆕 새 이슈 발견
    ↓
📝 이슈 문서 작성 (상태: 진행중)
    ↓
🔍 원인 분석
    ↓
🛠️ 해결 방법 구현
    ↓
🧪 검증
    ↓
✅ 해결 완료 (상태: 해결됨)
```

### 상태별 의미

- **✅ 해결됨**: 문제가 완전히 해결되고 검증됨
- **🔄 진행중**: 현재 작업 중
- **⏸️ 보류**: 현재는 해결할 수 없거나 우선순위가 낮음
- **❌ 미해결**: 해결 방법을 찾지 못함
